## 翻訳対応に関するパフォーマンス懸念

### 1. リクエスト毎の言語ファイル読み込み
- `app/Http/Middleware/HandleInertiaRequests.php` の `share()` で、毎リクエストごとに `lang_path("{$locale}.json")` と `lang_path("{$locale}/starter-kit.json")` を `file_get_contents + json_decode` しています。  
- これらは基本的にデプロイまで不変のリソースなので、APCu/Redis 等へのキャッシュや `once()` でのメモ化を行わないと、トラフィックが多い環境では I/O コストと JSON デコードコストが無視できなくなります。  
- フォールバックも常に同じ処理を繰り返すため、`locale === config('app.fallback_locale')` のケースではディスク読み込みが二重に走る点にも注意が必要です。

### 2. Inertia 応答ペイロードの肥大化
- `share()` から `lang` オブジェクト全体を Inertia の props に乗せており、クライアントへ全翻訳辞書を逐一送っています。  
- 翻訳キーが増えるほどレスポンス JSON が肥大化し、初回ロードや各遷移での転送量・パース時間が増大します。  
- ユーザーが参照しないページでも常に全キーを送る形なので、画面単位／チャンク単位の遅延ロードや、最低限のキーだけを共有する仕組みを検討しないと、特にモバイル回線などで体感速度が悪化します。
